There is no conflict between having a modeling language with deep semantics, both static and dynamic, and having an easy to use modeling language supporting a multitude of use cases and intuitively understandable.

The problems arise when it has poor semantics, but it is expected that the concrete syntax (notation), simple or rich, must be just intuitively understandable. We have different intuition.

IMHO we must design the language with a well thought out depth in semantics, but then give the users just as much of this as needed to not just relay on intuition.

A good example of this is the Essence language, specified using MOF. It is semi-formal. Some constructs are formally defined with clear semantics. There are however, identified semantic holes, which can to be filled by the user. Essence is used to describe practices, compose these to entire methods, activate these thanks to state progressions. The concrete syntax is very simple, using poker-sized cards allowing users to play many serious games. Gameplaying with cards is a very popular notation, and not new for Essence. What is new is that traditional game playing is done with cards with no defined meaning, which limits the number of games you can play, since they have to rely on being understood intuitively. The value of the Essence semantics is that it allows methodologists to design a huge number of easy to understand games which otherwise wouldn’t be possibly.

We have built Essence with a simple metaphor in mind: A car needs a motor, but to drive you don’t need to know much about the motor. Practices and methods are built with Essence, but to use these you don’t need to understand much of Essence. Similarly, a modeling language should have a deep, but balanced, semantics, but to use it you don’t need to know much about it.

拥有一个具有深层次语义（包括静态和动态）的建模语言，与拥有一个易于使用、支持多种用例且直观易懂的建模语言之间，并无冲突。

问题在于，当一种建模语言语义薄弱时，却期望其具体语法（表示法 notation）—— 无论简单还是丰富 ——必须仅凭直觉就能理解。我们有着不同的直觉。

依我之见，我们必须设计一种在语义深度上经过深思熟虑的语言，但随后只向用户提供恰到好处的部分，以避免仅仅依赖直觉。

Essence 语言就是一个很好的例子，它使用 MOF 进行规范定义。它是半形式化语言的。部分构造具有形式化的定义和清晰的语义。然而，同时也存在已识别的语义缺口，这些可由用户自行填补。Essence 用于描述一系列实践，将这些实践组合成完整的方法，并通过状态演进来激活这些实践。其具体语法非常简单，使用扑克牌大小的卡片，允许用户进行多种严肃游戏。用卡片玩游戏是一种非常流行的表示法，对 Essence 来说并不新鲜。其创新之处在于，传统游戏使用的卡片没有明确定义的含义，这限制了可玩游戏的数量，因为这些游戏必须依赖直觉来理解。Essence 语义的价值在于，它使方法论专家能够设计出大量易于理解的游戏，而如果没有这种语义，这是无法实现的。

我们在构建 Essence 时，心里有一个简单的隐喻（metaphor）：汽车需要发动机，但开车时并不需要对发动机有很深的了解。虽然实践和方法都是基于 Essence 的，但在使用时，并不需要对 Essence 有深入的理解。同样，建模语言应该具备深入但均衡的语义，但在使用时并不需要了解太深。

* a simple, more informal, modeling language intended for direct use by modelers which I think is what Grady and Steve are talking about.

From what I've seen so far I don't think the rigorous kernel meets the second need by itself. An open question, for me, is whether it could be used to define the second language. My doubts are whether the target community would benefit from the rigor and whether it would be too constraining.

Another question is whether it make sense to have just one simple language or to have different simple languages focused on specific purposes such as class modeling vs function modeling vs container modeling. Having our modeling paradigm so class-centered has held back OMG as the world has moved on from OO.

Intellectually it would be nice if all languages were built on a common kernel and the different models could be integrated and transformed. But that's the metamodel engineer in me speaking, and it's in danger of losing sight of the user base.

SysML is achieving major improvements while UML remains stagnant. My impression of the trouble is that there isn’t a consensus among the many UML RTF members nor the OMG in general about what should be done. I can see the following alternatives:

1. Keep UML as is

2. Proceed with fixing some of the 700+ UML issues and defer the rest to produce UML 2.6

3. Create an RFP for UML 3.0

4. Some combination of 2) and 3)

Creating an RFP is a lot of work. If we want an RFP, I suggest that we start with a discussion about what are the goals for what we want to achieve. Perhaps you can set aside a time slot in the ADTF agenda for this. In this discussion we might want to try to define a few use cases for how future UML users will use the new version of the language. Even a vague understanding of these goals and use cases will aid in the writing of requirements.

The following is a list of some goals for UML RFP that I may or may not agree with:

1. Migrate UML to KerML

2. Create a text language for UML similar to the SysML text language

3. Remove some features to obtain a simpler language and then migrate the remaining complex features into more powerful language(s)

4. Add features to UML to support functional programming

5. Add features to UML to support better composite data type definitions and data relations. This could be for databases, or it could be for arrays and vectors and matrices for a better mapping to MATLAB or to aid in basic deep learning algorithm descriptions, etc.

6. Add whatever features necessary to UML to obtain better support for describing ontologies

The above list seems daunting. Perhaps the RFP could just tackle the first 2 items above. It may be possible for items 4, 5, 6 be done in way the doesn’t break existing applications and thus we could claim that they could be done in a series of RTFs.

If we think we want UML 3.0 we need to discuss how close should UML follow SysML. For example: should the UML text language adopt the keyword for “part”, or should it continue to use “class”? Another example is that SysML v2 has established a pattern for the concepts of “def” verses “usage”. Should this be adopted, or should UML continue with the similar but different concepts of “class” and “instance”? Another example is the way SysML defines interface. The UML concept of interface is more traditional and has a better mapping to Java interface or Ada spec or an IDL interface. A UML interface may not be very close to a C++ header, but it is closer than SysML interface.
